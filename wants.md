# 持ち出し袋チェッカー

## 一行で言うと
防災リュック(持ち出し袋)の中身を登録し、賞味期限・電池切れ・消費期限を季節ごとに点検リマインドする「自宅備蓄の鮮度管理」アプリ。

## 想定ユーザー
- **直接ユーザー**: 防災備蓄をひととおり揃えたが、その後の鮮度管理が放置になっている個人・世帯
- **最終読者**: 同じ世帯の家族（共有して全員で備蓄状態を把握）

## 解決する課題
- 「備えたつもり」の防災備蓄が、いざという時に **賞味期限切れ・電池切れ・サイズ不適合** で使えない
- 既存代替手段:
  - 防災情報アプリ (NHK / Yahoo!防災等) — 災害情報の受信が主目的で、自宅備蓄の在庫鮮度管理機能は薄い
  - 汎用リマインダー / カレンダー — 品目ごとの期限管理に向かず、点検が形骸化
  - 紙の在庫チェック表 — 更新されず放置されがち
- 不満点 / 未解決の余地: 「自宅にある備蓄を品目単位で登録し、期限が近づいたら教えてくれる」点に特化したものが少ない。かつ**不安を煽らず淡々と点検できる**トーンのものが希少

## 主要ユースケース
1. **中身を登録する** — 持ち出し袋の品目(水・非常食・モバイルバッテリー・乾電池・常備薬の予備・現金・コピー書類等)を、写真または手入力で登録。期限・数量・保管場所を記録
2. **鮮度の点検通知を受ける** — 賞味期限/消費期限/電池の推奨交換時期が近づいたら通知(任意ON)。淡々とした「そろそろ点検どうぞ」のトーン
3. **季節点検をする** — 防災の日(9/1)や年2回などのタイミングで「まとめて点検」モード。チェックリスト形式で一気に確認
4. **家族で共有する** — 世帯メンバーで同じ備蓄リストを参照・更新(誰が補充したか分かる)
5. **買い替えリストを作る** — 期限切れ・不足分を買い物リストとして書き出し(印刷/共有)

## 競合・既存サービスとの差別化
- 既存サービス: NHK ニュース・防災 / Yahoo!防災速報 / 各種「防災リュック」物販サイト / 汎用在庫管理アプリ
- 差別化仮説: **災害"情報"アプリでなく「自宅の備蓄の鮮度管理」に特化**。物販でもなく、すでに持っている備蓄を"使える状態に保つ"ことだけにフォーカス
- AI 価値検証観点 (Q12.5(7) 相当): 汎用 AI チャットに「備蓄管理を手伝って」と頼むより、品目テンプレ・期限ロジック・点検リマインド・家族共有が構造化されている点が価値。MVP では AI は必須でない(将来、品目写真からの自動登録に Vision を任意採用)

## データ規模感
- 1 世帯あたり登録品目: 数十〜100 件程度
- ユーザー数: 個人開発の緩やかな立ち上がり想定(初期 〜数百)
- 画像(品目写真)を持つ場合は Cloudflare R2 へ。テキスト中心なら DB のみで軽量

## 想定同時利用者数
- 低い(点検は年数回〜月数回の利用パターン)。常時接続型でないため同時負荷は小さい

## 外部連携
- MVP: なし
- 将来候補: 自治体/公的機関の「推奨備蓄リスト」を参照テンプレとして取り込み(出典明記)。通知は Web Push でなくメール/アプリ内が無難(iOS Safari Push の不安定さ、charter §2.2 中毒性回避)

## 非機能要件（初期想定）
- 性能: 軽量(品目リストの CRUD が主)。シビアな要件なし
- 可用性: 個人ツール水準で十分(SLA 不要)
- セキュリティ: 登録するのは「自宅の持ち物リスト」で機微情報ではない。ただし保管場所等は世帯のプライバシーに関わるため、認証で保護しデータは本人/世帯に限定。常備薬の"予備の有無"程度は扱うが、病名・処方内容など機微医療情報は扱わない(C2 回避)

## 体制・予算・納期
- 1 人開発 / 初期コスト $0(無料枠厳守) / 開発期間 1〜2 ヶ月で運用開始

## 技術選好（preferences 由来 + 当該案の特性）
- フロント: Vite + React + TypeScript(preferences §2.1 採用2、shadcn/ui + Tailwind)
- バック: Vercel Functions(サーバーレス)
- DB: Neon (Postgres) — サービスごとに DB 完全分離(charter §0.2 / preferences §3.1)
- ORM: Drizzle
- ホスティング: Vercel (Hobby)
- 認証: Clerk(Free 10k MAU)。ゲスト/段階的認証を推奨(charter §1.1) — まず触れて、家族共有/複数端末/課金のタイミングで認証
- Storage: 品目写真を使う場合のみ Cloudflare R2
- 運用連携: service-hub 連携の標準 service-info エンドポイントを公開(perspectives O48、契約 SoT=service-hub)

## ローカル開発環境（初期想定）
- Vite dev server + Neon の dev ブランチ + Clerk development インスタンス
- スマホ実機確認は Windows/WSL2 のポートフォワード案内(/flow:release 手順参照)

## 外部 AI サービス組み込み（任意）
- 使う / 使わない: **MVP は使わない**
- 使う場合の用途・プロバイダ・差別化根拠: 将来、品目写真からの自動登録(食品ラベルの賞味期限 OCR / 物体認識)に OpenAI Vision 等を任意採用。コストは PWYW の従量課金で吸収

## 法務・コンプライアンス（初期想定）
- プライバシーポリシー: **必要**(Clerk 経由のメール等の個人情報を扱うため。perspectives O12 法令必須)
- 特商法表記: **有償提供(PWYW/買い切り)を公開する時点で必要**(国内有償、perspectives O14 法令必須)
- 業務固有規制: なし(防災"情報の正確性"を保証する立場でなく、ユーザー自身の備蓄管理ツールに徹する。公的避難情報の発信はしない)

## 収益モデル仮説 (charter §1 由来)
- monetization_hint: **PWYW**(主導線、charter §1.2 第一推奨)
- 詳細: 基本機能(品目登録・点検通知・1人利用)は無料。**家族共有 / 品目テンプレ拡張 / 買い替えリストPDF出力**を 300 円前後の一回課金 or PWYW でアンロック。「年に数回しか開かないけど、ちゃんと備えられて助かった」と感じたタイミングで"ぽちっ"と支援したくなる導線
- サブスクの場合の継続価値根拠 (charter §1.4): 該当なし(一回課金/PWYW を主導線とする。継続的リソース提供がないためサブスクは不採用 — C11 回避)

## 社会善観点 (charter §2 由来)
- social_good_score: **5**
- 寄与パターン: ユーザー個人 + 周囲(家族) + 社会(地域の防災レジリエンス向上)
- 価値観マッチ理由: 災害時に「備蓄が実際に使える」状態を保つことは、本人と家族の安全に直結し、地域全体の災害対応力(共助の前提となる自助)を底上げする。物販で不安を煽って買わせるのでなく、すでにある備えを活かす方向

## アンチパターン避けチェック (charter §2.2 由来)
- **不安を煽らない設計**: 災害の恐怖や「あなたは危ない」を煽らない。「そろそろ点検どうぞ」程度の淡々としたトーン。災害発生確率の脅し表示はしない
- **競争を煽らない設計**: 他世帯との備蓄比較・ランキングは作らない。自分の備蓄が整っているかだけを見る
- **中毒性を狙わない設計**: 通知は任意ON・低頻度(季節点検中心)。毎日開かせる仕掛けは作らない(年数回の利用で完結してよい設計)
- **プライバシー尊重設計**: 備蓄品目・保管場所は世帯内に限定。第三者提供・データ売買なし。位置情報は使わない(避難所連携は将来検討時も任意)

## 想定難所 / リスク
- 技術: 通知の確実性(メール/アプリ内通知の設計)。Web Push は iOS で不安定なため慎重に
- 運用/継続: **「年に数回しか開かない」ため継続利用の動機づけが最大の難所**。季節点検のリマインドと、開いたときの"片付く感"が鍵
- 入力負担: 品目登録の初期手間。テンプレ(標準的な持ち出し袋セット)プリセットで緩和
- 法務: 防災情報の正確性を保証しない旨の免責。避難判断はユーザー/公的情報に委ねる
- 市場: 「防災アプリ」は数多いが、備蓄"鮮度管理"特化はニッチ。逆に言えば既存と競合しにくいポジション
