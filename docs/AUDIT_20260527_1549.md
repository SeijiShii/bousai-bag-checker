### 3.2 High

#### [AUDIT-perspective-001] O48 service-info エンドポイントが api/ ルートに未配線
- **カテゴリ**: #4 観点反映 (O48 service-hub 連携、連携必須)
- **検出根拠**:
  - concept §1.3.2 / §6 / §4 で `GET /api/service-info` (service-hub が pull する運用指標エンドポイント) を **require** 観点 O48 として設計。
  - core ロジック `src/services/serviceInfo/handler.ts` (`handleServiceInfo`: トークン認可 401 + レート制限 429 + 集計 200) + `collectMetrics.ts` は実装済、7 unit test green。
  - **しかし `api/service-info.ts` (Vercel Function) が存在しない**。`api/` 配下は items/feedback/inspection/settings/shopping/stripe-webhook/tip/cron-expiry-check はあるが **service-info だけルートファイルが欠落**。
  - `grep -rn serviceInfo api/` = 0 件 → どの api ルートからも `handleServiceInfo` が import されていない。
  - 結果: **service-hub は実行時に `GET /api/service-info` を pull できない** (404)。require 観点が routing 層で未実装。
- **severity**: **High** (O48 連携必須エンドポイントの実行時未提供。CF-20260527-011 と同型: 設計+core 実装済だが endpoint 未配線)
- **修正が no-key Class A である根拠**: `api/_lib/composition.ts` に `getDb()` (Neon) があり、`ServiceInfoDeps` は db + metrics(=collectMetrics) + rateLimiter(InMemory 既定) + expectedToken(`SERVICE_INFO_TOKEN`)。**実 Clerk/Stripe/Resend は不要**。`SERVICE_INFO_TOKEN` は設定可能な共有シークレット。よって実キー無しで endpoint 配線 + 結合テストが可能。
- **推奨修正アクション**: `/flow:revise _shared/service-info` — `api/service-info.ts` を新設し、リクエストヘッダのトークンを読み `handleServiceInfo(deps, token, clientKey)` を呼ぶ Vercel Function を配線。deps は composition で db + collectMetrics + InMemoryRateLimiter + `process.env.SERVICE_INFO_TOKEN` を組み立て。結合スモークを追加。
- **§8 登録**: 登録推奨 (`--auto-register-issues` 未指定)。ただし `/flow:auto` §3.0c drift シューティングで即 dispatch するため §8 登録より実装を優先。
